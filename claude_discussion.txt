In the database layer:

CharField maps to VARCHAR(n) in most databases - a variable-length string with a maximum limit you specify

But here's the deeper question: Why does this matter for performance and storage?
Think about how a database works. When you set max_length=500 on a CharField, the database knows "this field will never be longer than 500 characters." When you use TextField, you're saying "this could be any length."

You're on the right track with memory allocation, but let me refine that a bit.
Storage-wise: Both VARCHAR and TEXT are actually pretty efficient in modern databases - they both only store the actual characters you put in them, not empty space up to the max. So a 10-character string in VARCHAR(500) only takes up ~10 characters of storage.
But here's where the real difference matters:
Indexing and performance. Databases can create indexes on columns to make searches faster. With a VARCHAR(30), the database can efficiently index the entire field. With TEXT fields, most databases either can't index them at all, or can only index the first N characters.
Think about it this way:

roast_level (VARCHAR 30) - you might search/filter by this: "show me all light roast beans"
description (TEXT) - you probably won't be doing SQL queries like "find all brew methods where description contains..."

So the rule of thumb: Use CharField for short, bounded data that you might search/filter on. Use TextField for longer, unbounded content that's mainly for display.
